# 2048 Game Implementation Plan

> **Goal:** Implement a working 2048 game as a Kali artifact window — grid-based, turn-based, keyboard input, configurable board size, neon visual theme, score tracking.
>
> **Status:** Implemented.
> **Last updated:** 2026-07-01.

## Architecture

Single-player 2048 running entirely in the frontend as a grid-based turn-based game. The `TwentyFortyEightGame` class (`kali-web/src/games/twenty-forty-eight/twenty-forty-eight-game.ts`) manages board state, tile merging, spawn logic, win/loss detection, and pause/resume lifecycle. A React `TwentyFortyEightView` component renders the board with a neon palette and handles keyboard input and universal overlays.

## Tech Stack

TypeScript, React, CSS Grid, existing Kali Toys core (`BaseGame`, `GameRegistry`, `GameWindow`), existing `GameStatus` / `ActionType` / `GameCommand` constants.

## Global Constraints

- Extend `BaseGame` from `src/games/core/base-game.ts`.
- Register via `GameRegistry.register(GameType.TWENTY_FORTY_EIGHT, TwentyFortyEightGame)`.
- Use existing constants; never redefine strings.
- State lives in the game class; the view reads `game.getState()` and triggers React re-renders only when status/score change.
- Build passes `npx tsc --noEmit` with zero errors.

**Behavior:**

- Handle `UP` / `DOWN` / `LEFT` / `RIGHT` moves.
- Merge adjacent equal tiles; each merge adds the merged value to the score.
- Spawn one new tile after each successful move.
- Detect win when any tile reaches `2048`.
- Detect loss when no empty cells and no merges are possible.

```typescript
interface BoardData {

  bestTile: number;
  moves: number;
  won: boolean;
  over: boolean;
  lastSpawned: { row: number; col: number } | null;
  mergedIds: number[];
}
```

## Lessons learned (post-implementation)

### 1. Animation state must be captured *before* the render that consumes it

Do not update the "previous board" ref during the same render pass that uses it. If `animRef.current` is overwritten while `prevPositions` is being computed, the previous position is lost and tiles appear to teleport or "repeat" the last move.

**Correct pattern:**
```typescript
const shouldAnimate = !animRef.current || !data || !cellsEqual(animRef.current.cells, data.cells);
const prevPositions = shouldAnimate && data ? buildPositionMap(animRef.current?.cells ?? []) : {};

useEffect(() => {
  if (shouldAnimate && data) {
    animRef.current = data;
  }
}, [shouldAnimate, data]);
```

### 2. Guard against repeated pop animations on re-renders

A tile that is already on screen should not re-run its spawn animation every time the component re-renders. Use a local `initializedRef` inside the animated tile so the pop/scale effect runs only the first time the tile is mounted.

### 3. Animate only when the board actually changed

Use a deep comparison of the board cells (including tile IDs and values) to decide whether to calculate previous positions. Comparing only `score`/`moves` or relying on the engine's `prevPositions` field proved insufficient because incidental renders (focus changes, clicks, drags) could re-trigger the animation from stale data.

### 4. CSS Grid tracks must be clamped to avoid layout deformation

Using `repeat(n, 1fr)` lets tile content expand the rows. Use `repeat(n, minmax(0, 1fr))` so every track has a minimum size of 0 and stays perfectly square regardless of text size.

### 5. Include padding when calculating fixed cell size

If the grid has padding, the cell size formula must subtract `2 * padding + (n - 1) * gap`. Otherwise the tile layer will not align with the background grid.

### 6. Fix the header dimensions so text growth does not expand the board

Score/movement text grows as the game progresses. If the header is allowed to expand, it pushes the grid and makes the whole board appear to grow. Give the header a fixed height, put the numeric value on a second line, and size the score/move boxes explicitly.

### 7. Turn-based games may not need pause-on-focus-loss

The generic `GameWidget` paused any active game when focus was lost. For a turn-based game like 2048 this caused unnecessary re-renders that interfered with animation state. Pausing on focus loss should be restricted to real-time games (e.g. Snake).
